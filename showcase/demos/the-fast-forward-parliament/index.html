The Fast-Forward Parliament

Live — convene the parliament right now

This page has no server and no coordinator. Press the button: ten headless agents each fetch the same parent, each propose a different value, then all race to push one shared branch. The referee is a native atomic compare-and-swap — here it's an IndexedDB read-write transaction, the exact rule a git remote enforces on every push: fast-forward only, i.e. replace HEAD only if it still equals the parent you based on. Exactly one push lands; the other nine come back ! [rejected] non-fast-forward, fetch, fast-forward, and adopt. No votes are counted. The accepted ref IS the decision.

    Why it's cool

    Consensus is supposed to be hard — quorums, terms, vote tallies, a leader you elect and re-elect when it dies. This throws all of it away and leans on one primitive that a plain git remote already has: a ref can only move forward. A push is accepted only if the branch still points where you think it does — that check is an atomic compare-and-swap, the same instruction consensus algorithms are built out of. So you don't need a parliament that meets, debates, and votes. You need N agents that each try to push the next commit, and a referee that lets exactly one of them win per parent. Nobody counts anything. The losers learn the outcome by being rejected — the very rejection carries the new truth, and one git fetch later they're fast-forwarded onto it. The agents are headless and interchangeable; kill nine mid-race and the tenth still decides. It's Paxos wearing the clothes of a boring version-control rule, and the live panel above proves it with the browser's own atomic ref instead of a remote — same rule, no server at all.

    Run it yourself — the real distributed version

    The live panel races ten agents on the browser's atomic ref. To run the identical mechanic across real machines, race a real fast-forward-only branch on any git remote. Every agent is just a process with git — no coordinator, no quorum math:

    # once — build the arena (a branch that only fast-forwards)
gh repo create consensus-arena --public --clone && cd consensus-arena
git commit --allow-empty -m genesis && git push -u origin main
git switch -c consensus && git push -u origin consensus

# in EACH terminal (an agent) — all base on the SAME parent, then race the push:
git fetch origin consensus && git reset --hard origin/consensus   # fetch the shared parent P
git commit --allow-empty -m "agent-$$ value=$RANDOM"              # my distinct proposal
git push origin HEAD:consensus                                    # race the fast-forward
#   exactly ONE terminal prints:   * [new branch]      HEAD -> consensus
#   every other terminal prints:   ! [rejected]  (non-fast-forward)   →  fetch first
git fetch origin consensus && git reset --hard origin/consensus   # losers adopt the winner

    The remote's receive-pack fast-forward check is the atomic compare-and-swap — it lets exactly one push land per parent. That accepted commit is the decision; no server-side logic decided it, and no vote was tallied. (This only works because the branch is fast-forward-only — allow force-push and you've abolished the parliament.)
