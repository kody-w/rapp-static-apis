One String, Two Universes

The one string

One content-address — kody-w/fn@ce06d0379462 — names an exact 2,417-byte compute cell. An AI dereferences it as an MCP tool in Node. A phone dereferences it from a QR in a browser. Neither runtime trusts a host: each fetches the bytes, recomputes SHA-256, and refuses unless it matches the pin. Same string, same bytes, two universes.

          ┌───────────────── /resolver ─────────────────┐
"one string"  kody-w/fn@ce06d0379462   →  owner · name · sha8
          └──────────────────────┬──────────────────────┘
                                 │  the one pinned frame (/fn twin cell)
                 ┌───────────────┴────────────────┐
        NODE DOOR (/mcp)                   BROWSER DOOR (/track · QR)
   AI calls ping()/dataverse_search   phone scans → run.html sandbox
   shim: fetch→verify sha8→import→run  fetch→verify sha8→run (no install)
                 └────────── verify-before-exec ──────────┘
              flip one byte → BOTH refuse, same instant

    The Node door — what the AI runs (MCP)

    Point Claude Desktop / Cursor / the Copilot CLI at the shipped static-MCP shim. It carries zero per-tool logic — it reads a static catalog and, on every tools/call, does the same verify-before-exec you just watched:

    { "mcpServers": { "rapp": { "command": "node", "args": [
    "shim.mjs", "--catalog",
    "https://raw.githubusercontent.com/kody-w/rapp-static-apis/main/mcp/tools.json" ] } } }

    The catalog binds the ping tool to the same frame the browser door ran:

    "ping": { "cell": "twin", "export": "echo",
          "sha8": "ce06d0379462",
          "pin_path": "versions/twin/ce06d0379462.mjs" }

    And the shim's verify line is the Node twin of runVerifiedCell's hash check — refuse first, run second:

    const got = createHash("sha256").update(bytes).digest("hex").slice(0,12);
if (got !== b.sha8)
  throw new Error(`verify-before-exec FAILED for ${name}: ${got} != pinned ${b.sha8}`);
const mod = await import("data:text/javascript;base64," + …);   // exact verified bytes
return await mod[b.export](args);

    Why it's cool

    Trust normally comes from where code runs — a server you log into, a vendor you pay, an app store that vetted the binary. Here trust comes from what the bytes hash to, so it survives crossing between worlds that share nothing else. An autonomous agent with no screen and a stranger's phone with no account reach for the same short string, pull the same bytes, and get the same proof — one in Node over stdin, one in a browser sandbox from a scan. Nothing in the middle can lie to just one of them: corrupt the frame in a fork and the Node shim throws verify-before-exec FAILED while the browser throws refused:true, at the same instant, for the same 12 hex characters. Capability becomes a coordinate, not a location.

    Run it yourself

    1. The AI door. Drop the config above into your MCP host and restart it. Ask it to call ping({message:"hi"}) — the round trip comes back verified — then dataverse_search({q:"RAPP"}), which runs the same frame's live() export and fetches the static Dataverse twin from inside the sandbox.
    2. The human door. Open /track, point a spec at a pinned cell, hit Make QR (the zero-dependency encoder in qr.mjs). Scanning it loads run.html, which verifies the sha8 inside a sandboxed iframe before it runs — the exact same verify-before-exec, triggered by a camera instead of a tool call. Print it anywhere.
    3. The address. Both doors dereference the one string through /resolver — expand("kody-w/fn@ce06d0379462") supplies the host so no human ever types it.

    Local: python3 -m http.server in the repo root, then open localhost:8000/showcase/demos/one-string-two-universes/ (crypto.subtle needs a secure context — localhost or https, not file://).
