    Why it's cool

    Every "is this code safe?" answer you've ever trusted came from something you also had to trust — a server, an antivirus vendor, an app-store reviewer, a TLS cert for a host that could still ship you different bytes tomorrow. This has none of that. The code is content-addressed: its name is its SHA-256. Before a single line executes, your browser re-hashes the exact bytes it fetched and compares them to the pin. Match, and it runs. One byte off — a smuggled fetch(), a swapped API key, a hostile edit at the CDN — and it refuses, offline, with no one to phone home to.

    That's the inversion: you stop trusting the host and start trusting the hash. A doubter doesn't have to take your word for it. They flip a byte themselves, in their own browser, and watch it get caught — the red button above is that dare, made real.

    Run it yourself

    1. Click ✓ Run verified: the hash step goes green (✓ … matches the pin) and the price prints — the code ran because it was proven intact.
    2. Click 🪤 Tamper & run: same fetch, but one byte of the recomputed hash is flipped to simulate an altered file. The hash step goes red (✗ … ≠ pinned) and verify-before-exec REFUSED — the tampered code never runs.
    3. Want the real thing? Open /track, edit one character of versions/extract/<sha8>.mjs without re-pinning, and reload: the live runner refuses it exactly the same way — no simulation.

    Local: python3 -m http.server in the repo root, then open localhost:8000/showcase/demos/the-byte-trap/ (crypto.subtle needs a secure context — localhost or https, not file://).

    For agents 🤖

    Reusable pattern:

    Call runVerifiedCell({ base, cell, fn, args }) whenever you're about to import code from a host you don't control. It resolves the pin, re-hashes the fetched bytes, throws { refused:true } on any mismatch, and only then imports the exact verified bytes. Verify-before-exec, byte-for-byte, before your agent runs anything.
