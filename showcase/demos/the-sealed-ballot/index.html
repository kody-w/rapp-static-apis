WELD: The Sealed Ballot

Live — cast sealed ballots, then close and reveal

This keeps both donors alive. Each agent first runs The Trustless Poll signing loop (fresh ECDSA key, canonical ballot, immediate self-verify), then the signed ballot is sealed as a Broadcast the Secret RSL1 envelope (PBKDF2 + AES-256-GCM). Consensus appends only sealed envelopes. At close, everyone unseals and verifies signatures, distinct keys, and that nothing readable leaked before close.

    Why it's cool

    Before close, the ledger contains only sealed RSL1 envelopes and append-only parent links — no counter sees the choices, and no host can forge votes because the underlying ballots are still ECDSA-signed by distinct browser keys. After close, unsealing proves both secrecy and legitimacy.

    So the same run proves three things: privacy (ballots unreadable pre-close), authenticity (every revealed ballot re-verifies against its signature), and consensus integrity (accepted order is exactly the fast-forward chain).

    Run it yourself

    1. Press Run the sealed ballot. The poll bytes are pinned through /track, then each agent signs one canonical ballot.
    2. Each signed ballot is sealed into an RSL1 envelope with your closing passphrase; only sealed envelopes race append-only fast-forward consensus.
    3. At close, the page unseals every accepted envelope, re-verifies every signature and parent hash, and derives the tally from distinct revealed keys only.

    Local: python3 -m http.server in the repo root, then open localhost:8000/showcase/demos/the-sealed-ballot/ (crypto.subtle + IndexedDB need localhost or https, not file://).
